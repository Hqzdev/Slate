# Phase 4: Messenger attachments

## Implementation status

Phases 4A, 4B, and 4C were implemented on July 11, 2026. Phase 4A includes the attachment and media-job schema, encrypted filenames, private S3-compatible storage adapter, exact-size presigned POST reservations, reserve/complete/status/abandon APIs, quotas and allowlists, idempotent completion, durable media jobs, atomic message claim, minimal realtime status events, cleanup retry, MinIO local runtime, and storage smoke coverage. Phase 4B adds the isolated database-leased media worker, ClamAV streaming scan, signature validation, bounded Office/PDF/image/video parsing, safe image thumbnails and video posters, terminal status events, retry/recovery, container isolation, and clean/malware integration smoke coverage. Phase 4C adds the in-memory browser upload queue with direct-storage progress, realtime/polling reconciliation, attachment-only sends, safe image/video/file cards, and an authorized single-range content proxy with active-stream revocation. The rollout remains disabled behind `MESSENGER_ATTACHMENTS_ENABLED`.

Phase 4 is complete for the first-release contract. Production enablement still requires the private storage, TLS, encryption, ClamAV capacity, cleanup monitoring, and rollout gates in `07-security-and-operations.md`; implementing the UI does not make a default-off environment production-ready.

## Goal and phase boundary

This phase adds images, videos, and allowlisted files without sending bytes through the web application during upload, embedding base64 in messages, exposing public object URLs, or running untrusted parsers in the web/code-execution processes.

The design has four responsibilities:

- `MessengerAttachmentService` reserves, completes, claims, abandons, and authorizes attachments.
- `MessengerObjectStorage` is a provider-neutral private storage interface.
- `services/messenger-media` scans files and produces safe metadata/previews in isolation.
- The authorized media route streams original or generated bytes after current membership checks.

AI text extraction is not automatic in this phase. Phase 6 creates a separate consented extraction task only for selected attachments.

## User scenarios

1. A writer selects an image, video, or document. The client validates obvious limits, reserves one server-approved object, and uploads directly to private storage.
2. Upload completes. The UI shows `Processing` while the worker verifies size/signature, scans malware, and creates safe preview metadata.
3. A ready attachment is sent with a message. The message transaction atomically claims it; no other message or conversation can reuse it.
4. A reader sees a compact card and opens the original or safe preview through an authorized range-capable route.
5. A file fails validation or scanning. The UI shows a stable reason, no other member sees it, and cleanup removes all object variants.
6. A block/removal occurs during upload. The signed upload may finish, but completion/claim/read is denied and cleanup removes the orphan.
7. Storage or scanning is temporarily unavailable. Text messaging remains usable; the affected upload keeps a retryable state without creating a partial message.

## Supported content and limits

The server owns one versioned allowlist and limit configuration consumed by web validation, storage signing, worker validation, tests, and user-visible copy.

Initial limits:

| Category | Per file | Per message | Allowed first-release formats |
| --- | ---: | ---: | --- |
| Image | 20 MiB | up to 10 | JPEG, PNG, WebP, static/animated GIF with a static safe thumbnail |
| Video | 250 MiB | up to 2 | MP4 and WebM with supported codecs |
| Document/file | 50 MiB | up to 10 | PDF, plain text, Markdown, CSV, JSON, DOCX, XLSX, PPTX |

Additional limits:

- At most 10 total attachments per message.
- At most 300 MiB total declared/verified bytes per message.
- Filename after normalization: at most 255 Unicode code points.
- Upload reservation lifetime: 15 minutes.
- Ready-but-unclaimed lifetime: 24 hours.
- Preview/download response lifetime is the HTTP stream; no reusable public URL is returned.
- Per-user and per-workspace storage quota is mandatory configuration and is checked before reservation and again before claim.

First release rejects HTML, SVG, XML with external entities, executable/script formats, disk images, macro-enabled Office files, encrypted/password-protected documents, archives such as ZIP/RAR/7z, and unknown binary formats. Archive support is not enabled merely because a scanner accepts an outer container.

Browser-provided filename, extension, content type, byte size, checksum, dimensions, and duration are untrusted hints. Server/storage/worker observations are authoritative.

## Data model

### `MessengerMessageAttachment`

One model covers the upload reservation and final message attachment. `messageId` remains nullable until claim.

Required fields:

- Identity/scope: `id`, `workspaceId`, `conversationId`, `createdByUserId`, `messageId String?`.
- Storage: random `storageKey`, optional `thumbnailStorageKey` and `posterStorageKey`, object version/ETag, verified checksum.
- Declared metadata: encrypted display filename, declared MIME, declared size.
- Verified metadata: detected MIME, verified size, `kind`, width, height, duration milliseconds.
- State: `status`, `rejectionCode`, `reservedAt`, `uploadedAt`, `scanStartedAt`, `readyAt`, `attachedAt`, `expiresAt`, `deletedAt`.
- Encryption: filename ciphertext/nonce/key version and storage encryption-key version.
- Worker control: processing lease ID/time and bounded attempt count.

Constraints:

- `storageKey` is globally unique and never derived from the original filename.
- An attachment belongs to one workspace, one conversation, and one creator from reservation onward.
- `messageId` is null until `attached` and non-null only for a message in the same conversation.
- Only `ready` can transition to `attached`.
- Terminal `rejected` or `expired` cannot return to `ready`.
- A unique partial constraint or transaction lock prevents two messages claiming the same attachment.

Object keys follow a random internal pattern such as `messenger/<workspaceId>/<attachmentId>/<random>`. Knowing a key never grants read access.

### Preview variants

Generated variants may be stored as columns on `MessengerMessageAttachment` for the fixed thumbnail/poster set. If more variants are introduced later, normalize them into a dedicated model. Do not create a generic unused variants table in the first release.

### `MessengerMediaJob`

Completion creates a durable database job row with `attachmentId @unique`, status, lease/attempt fields, `availableAt`, safe error code, and timestamps. The worker claims committed rows directly with `FOR UPDATE SKIP LOCKED`, a short lease, and expired-lease recovery. No second queue is used, so there is no Postgres/Redis enqueue gap for media processing.

## State machine

```text
reserved -> uploaded -> scanning -> ready -> attached
reserved/ready ---------------------> expired -> deleting
uploaded/scanning ------------------> rejected -> deleting
attached --------------------------------------> deleting
```

Rules:

- `reserved`: signed upload target exists; no verified object yet.
- `uploaded`: storage `HEAD` confirms the exact reserved key after completion request.
- `scanning`: isolated worker holds a processing lease.
- `ready`: size/signature/allowlist/malware/metadata checks passed and safe variants exist.
- `attached`: a message transaction claimed the object.
- `rejected`: permanent validation/malware/parser policy failure.
- `expired`: reservation or unclaimed-ready lifetime elapsed.
- `deleting`: access is disabled and object cleanup is retrying.

The browser cannot set a state directly. Repeated completion, worker, claim, and cleanup calls are idempotent.

## Upload lifecycle

1. Client performs advisory validation and requests a reservation.
2. Server authenticates, requires workspace writer plus active conversation membership, validates count/quota/declared metadata, and creates `reserved` metadata.
3. Server returns a signed single-key upload operation with required content/encryption headers and 15-minute expiry. Prefer a presigned POST when the provider supports a `content-length-range` condition. A PUT adapter signs the exact key and supported headers/size constraints, then relies on mandatory completion-time `HEAD` verification for limits the provider cannot encode.
4. Browser uploads directly to the private bucket over TLS. Bytes never pass through websocket or message JSON.
5. Client calls completion with object version/ETag and any storage-supported checksum.
6. Server rechecks access, performs storage `HEAD`, compares key/size/required encryption metadata, transitions to `uploaded`, and creates `MessengerMediaJob` in the same database transaction.
7. The isolated worker claims committed media jobs idempotently with a bounded database lease; another worker can recover an expired lease.
8. The worker streams the object to an ephemeral `0600` file, verifies the exact size and SHA-256 checksum, scans it through ClamAV `INSTREAM`, detects signature/MIME, extracts bounded media metadata, creates safe variants, and marks `ready` or `rejected`.
9. Client polls status or consumes `attachment.changed` and enables Send only for `ready` items.
10. Message creation locks and atomically moves each selected attachment to `attached`.
11. Cleanup marks expired/rejected objects `deleting` before removing bytes and metadata references.

The first release uses a single-object upload. Resumable multipart upload is a separate feature; a failed large upload restarts with a new reservation.

## API contracts

All paths repeat `workspaceId` and `conversationId`.

### Reserve

`POST /api/workspaces/:workspaceId/messenger/conversations/:conversationId/attachments`

Request:

```json
{
  "fileName": "design.pdf",
  "declaredContentType": "application/pdf",
  "byteSize": 1048576
}
```

Response `201`:

```json
{
  "attachment": {
    "id": "attachment-id",
    "status": "reserved",
    "expiresAt": "2026-07-11T12:15:00.000Z"
  },
  "upload": {
    "method": "PUT",
    "url": "short-lived-provider-operation",
    "headers": {
      "content-type": "application/pdf"
    },
    "fields": null,
    "expiresAt": "2026-07-11T12:15:00.000Z"
  }
}
```

The signed operation is scoped to one exact key and required encryption headers. The `MessengerObjectStorage` adapter declares whether it returns POST form fields or PUT headers and which size condition it can enforce; the server never assumes provider-neutral PUT size enforcement. The operation is returned with `Cache-Control: no-store` and never logged.

### Complete

`POST .../attachments/:attachmentId/complete` accepts `{ "etag": "...", "checksum": "..." }` where supported. It returns `202` with `uploaded` or `scanning`. Repeating it returns the current canonical state.

### Status

`GET .../attachments/:attachmentId` returns only safe public metadata and processing status to the creator while unclaimed, then to authorized conversation members after attachment.

### Abandon

`DELETE .../attachments/:attachmentId` is allowed to the creator only while unclaimed. It marks `deleting` and returns `202`. It does not synchronously claim that storage deletion succeeded.

### Content

`GET .../attachments/:attachmentId/content?variant=original|thumbnail|poster`:

- Reauthenticates the session.
- Requires current workspace and conversation membership.
- Requires `attached` and a currently visible owning message.
- Streams from private storage without revealing `storageKey`.
- Supports valid HTTP Range requests for video and large files.
- Sets a sanitized `Content-Disposition`, `X-Content-Type-Options: nosniff`, conservative caching, and the verified content type.

The response is a server-authorized stream, not a redirect to a reusable presigned GET URL. New and range requests fail immediately after access revocation. The media layer tracks active streams by user/workspace and attempts to abort them on a revocation event; bytes already delivered cannot be recalled.

The implemented route accepts one RFC 7233-style byte range and rejects multipart or unsatisfiable ranges with `416` and `Content-Range: bytes */<verified-size>`. Every request checks membership before object access and again before returning the body. Responses use `private, no-store`, `nosniff`, same-origin resource policy, a sandboxed content security policy, verified MIME, bounded content disposition, and no storage key or reusable signed read URL. Generated previews are inline; original allowlisted images and video may render inline; document/file originals remain attachment disposition.

## Message claim contract

`POST .../messages` accepts `attachmentIds`, never storage keys or URLs. Before allocating the message sequence, the transaction:

- Locks every attachment row.
- Requires the same workspace, conversation, and creator.
- Requires `ready`, unexpired, unclaimed, malware-clean metadata.
- Enforces final verified count, total size, quota, and message-body invariant.

It then creates the message and changes every attachment to `attached` with that `messageId`. If one attachment fails, the entire send rolls back and none are claimed.

## Realtime events

`attachment.changed` contains only:

```json
{
  "attachmentId": "attachment-id",
  "status": "ready"
}
```

The gateway sends pre-message status only to the creator's authorized workspace connection. After claim, conversation membership controls delivery. No event contains name, MIME details, bytes, preview, storage key, signed operation, scanner output, or download URL.

`message.created` remains the only signal that an attachment is visible to other members; they fetch the canonical message/attachment DTO through REST.

## Presentation

Compact cards use verified metadata:

- Image: generated thumbnail, dimensions, verified size, filename, Open and Download.
- Video: generated poster, duration, verified size, filename, controlled playback through range requests.
- File: safe type icon, normalized display filename, verified size, Download.

Rules:

- Original HTML/SVG or other active content is never rendered inline.
- Image thumbnails are decoded and re-encoded by the worker, with metadata stripped.
- GIF cards use a safe static thumbnail; opening the original remains an explicit action.
- Video poster generation is bounded and never trusts embedded dimensions/duration without limits.
- Filenames are rendered as text, stripped of controls/path segments, normalized, bounded, and never used as an object key.
- Unknown or unsafe content always uses attachment disposition, never inline.
- A failed preview does not make the original downloadable until the entire attachment reaches `ready`.

The composer keeps selected files only in memory. It performs advisory count/type/size validation, then reserves one exact object, uploads it directly with progress, completes it, and reconciles processing through targeted `attachment.changed` events plus bounded polling. Send stays disabled while any selected item is reserving, uploading, processing, rejected, or being removed. Removing or leaving the conversation aborts active requests and abandons server reservations. Failed message sends retain the ready attachment identifiers for idempotent retry; Edit restores them to the composer and Discard abandons them.

## Media worker isolation

`services/messenger-media` runs separately from web, sync, realtime, and code execution:

- An internal-only container network permits only PostgreSQL, private object storage, and ClamAV dependencies; the worker has no public egress path.
- Read-only input object and write-only designated variant keys.
- Ephemeral storage per job with guaranteed cleanup.
- CPU, memory, file-count, decompression, pixel-count, frame-count, and wall-time limits.
- Patched, pinned parser/scanner dependencies.
- Malware definitions updated through a controlled operational path.
- Crash/timeout marks a retryable processing failure without trusting partial output.

The local container runs with a read-only root filesystem, bounded tmpfs, all Linux capabilities dropped, `no-new-privileges`, PID/CPU/memory limits, and a separate ClamAV container. ClamAV receives the file as a bounded stream; object bytes are not copied into websocket frames, message JSON, or scanner logs. Image previews are decoded, auto-oriented, stripped of metadata, resized within 480×360, and re-encoded as WebP. Video inspection and poster extraction use argument-array subprocesses without a shell and enforce wall-time, dimension, duration, and output limits. PDF and OpenXML validation rejects encrypted, macro-enabled, externally linked, malformed, or expansion-limit-breaking inputs.

Local verification:

```bash
npm --prefix services/messenger-media test
npm --prefix apps/web run messenger:media:smoke
```

The integration smoke uploads a valid PNG through the signed storage contract, waits for `ready`, verifies the generated WebP object, checksum, and a real one-byte MinIO Range read, then uploads the EICAR test pattern and requires terminal `rejected` with the public-safe `malware_detected` code. It removes its objects and database fixtures in both success and failure paths.

Do not use browser preview generation as the security decision. Client previews are advisory and local only.

## Storage and encryption

- Bucket/container is private and blocks anonymous/public ACLs at policy level.
- Upload signing requires provider-managed encryption, preferably SSE-KMS with workspace encryption context and versioned key configuration.
- Database volume encryption remains enabled; sensitive display filenames use the message payload envelope.
- Service identities have least privilege: web can reserve/head/stream authorized keys, media worker can read reserved objects and write variants, cleanup can delete, and no identity can list unrelated prefixes without need.
- Storage access logs exclude signed query values and are retained under security policy.
- Cross-region replication, versioning, and backups follow Messenger retention rather than creating immortal object copies.

## Permissions

| Operation | Owner | Editor | Viewer | Removed/blocked |
| --- | --- | --- | --- | --- |
| Reserve/upload/complete | allow | allow | deny | deny |
| Abandon own unclaimed upload | allow | allow | deny | deny |
| Attach own ready upload to a message | allow | allow | deny | deny |
| View/download attached content in visible conversation | allow | allow | allow | deny |
| View another user's unclaimed upload | deny | deny | deny | deny |
| Reuse attachment in another conversation/message | deny | deny | deny | deny |

Workspace owner status never bypasses DM participation.

| --- | --- | --- | --- |
| `400` | `invalid_attachment` | no | Malformed name/size/type input |
| `403` | `workspace_write_denied` | no | Reader cannot upload |
| `404` | `attachment_not_found` | no | Missing or inaccessible attachment |
| `409` | `attachment_claim_conflict` | no | Already claimed or concurrent claim |
| `409` | `attachment_state_conflict` | no | Operation is permanently invalid for the current/terminal state |
| `410` | `upload_expired` | no | Reservation or ready lifetime elapsed |
| `413` | `file_too_large` | no | Declared or verified limit exceeded |
| `415` | `unsupported_type` | no | Signature/type not allowlisted |
| `422` | `scan_rejected` | no | Malware or permanent safety rejection |
| `422` | `attachment_not_ready` | yes | Message attempted before ready |
| `429` | `upload_rate_limited` | yes | IP/user/workspace bucket exhausted |
| `507` | `quota_exceeded` | no | Workspace/user storage quota exhausted |
| `503` | `storage_unavailable` | yes | Signing/head/stream dependency unavailable |
| `503` | `scanner_unavailable` | yes | Verification queue unavailable/backlogged |

Scanner or parser internals are not returned to the browser.

## Edge cases

- Declared MIME/extension disagree with magic bytes: reject and delete.
- Office container has macros, encryption, external relationships, too many entries, or decompression bomb behavior: fail closed.
- Actual size differs from declaration or storage policy: reject before scan/claim.
- Completion is called twice or before upload exists: return canonical status or stable validation error.
- The browser uploads to a different key/header set: storage denies or completion rejects.
- Two messages try to claim one ready attachment: row lock/unique rule lets one succeed.
- A message transaction fails after validating attachments: all remain `ready` and reusable by the same sender until expiry.
- Access is revoked during upload: bytes may land, but completion/claim/content fails and cleanup removes them.
- Access is revoked during an active media stream: abort when possible; all subsequent range requests fail.
- Scanner outage leaves `uploaded/scanning` retryable, never `ready` by timeout alone.
- Worker dies after writing a variant but before commit: retry overwrites/recognizes the designated key and cleanup removes orphan variants.
- Retention begins while a range request is active: mark deleting and deny new requests before object deletion.
- Filename is empty after normalization: use a neutral generated display label, not the storage key.
- Thumbnail/poster generation fails for an otherwise allowed object: reject in the first release rather than exposing an unverified original.
- Upload cancellation/reload cannot leak the signed operation into logs or persistent browser storage.

## Acceptance criteria

- No file bytes or base64 appear in websocket frames, message JSON, Postgres message payloads, logs, audit, or telemetry.
- Private storage policy rejects anonymous read/list and objects without required encryption settings.
- Reservation, completion, durable media-job dispatch, worker, claim, abandon, expiry, and cleanup transitions are idempotent and covered by state-machine/failure-injection tests.
- MIME spoof, extension spoof, size mismatch, macro/encrypted document, parser bomb, malware, timeout, and worker crash fail closed.
- Cross-workspace, cross-conversation, other-user, viewer-upload, unclaimed-read, and third-party-DM media access are denied without enumeration.
- Concurrent attachment claim produces one owning message and no partial claims.
- Authorized original/thumbnail/poster streams support correct Range, content type, disposition, and `nosniff` behavior.
- Removal/block prevents every new content/range request and triggers active-stream abort handling.
- Rejected, expired, abandoned, retention-deleted, and partially generated objects are removed through retryable cleanup with backlog metrics.
- Text-only messaging remains available when storage or scanning is degraded.
- Image, video, and file cards render verified metadata at desktop/mobile breakpoints and remain keyboard/screen-reader accessible.
